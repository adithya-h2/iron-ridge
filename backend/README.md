# Iron Ridge Intelligent Sales Workflow

> Enterprise-grade Multi-Agent AI Sales Operating System for Iron Ridge Fire Apparatus

---

## Overview

Iron Ridge Intelligent Sales Workflow is not a chatbot. It is an **AI workforce** — a team of
six autonomous AI agents and one human principal, each playing a defined business role inside
the sales pipeline of Iron Ridge Fire Apparatus.

Every customer progresses through a structured pipeline:

```
Lead → Qualification → Requirements → Pricing → Approval → Order → Production → Delivery
```

The **Deal** is the primary business entity. Every agent reads from and writes to a Deal. Agents
never call each other directly. Instead, they publish `WorkflowEvent` records consumed by
**n8n** (external orchestration), which decides which agent to invoke next.

## Architecture

This system follows **Clean Architecture** with strict layer separation:

```
HTTP Request
    │
    ▼
API Route (app/api/routes/)          ← HTTP boundary only
    │
    ▼
Service Layer (app/services/)        ← ALL business logic
    │
    ▼
Repository Layer (app/repositories/) ← ALL database access
    │
    ▼
SQLAlchemy Models (app/models/)      ← Schema definitions
    │
    ▼
PostgreSQL Database
```

### Event-Driven Agent Orchestration

```
Agent.execute()
    │
    ├── reads Deal context
    ├── reasons with LLM
    ├── writes structured output
    ├── updates Deal stage
    │
    ▼
WorkflowEvent published
    │
    ▼
n8n webhook listener triggers
    │
    ▼
n8n invokes next agent endpoint
    │
    ▼
Slack notification → Victor (approval gate only)
```

## Engineering Principles

1. **Clean Architecture** — strict layer separation enforced structurally
2. **Repository Pattern** — all DB access encapsulated
3. **Dependency Injection** — via FastAPI `Depends()`
4. **SOLID Principles** — Single responsibility per class
5. **Async First** — all I/O is async (SQLAlchemy async, httpx)
6. **Event-Driven** — agents communicate only via `WorkflowEvent`
7. **Audit Everything** — all mutations create `AuditLog` entries
8. **Prompt Injection Protection** — all user input sanitized before LLM calls
9. **No Secrets in Code** — all credentials via environment variables

### Agent Endpoints (n8n compatible)

Agent routes accept **form-encoded POST** bodies and return flat JSON for n8n `$json.*` expressions.

| Endpoint | Key response fields |
|----------|---------------------|
| `POST /agents/marty` | `deal_id`, `lead_score`, `status`, `current_agent` |
| `POST /agents/lisa` | `qualified`, `status`, `current_agent`, `lead_score` |
| `POST /agents/neil` | `status`, `requirements`, `current_agent` |
| `POST /agents/paul` | `quotation_id`, `quotation_generated`, `approval_status` |
| `POST /agents/sally` | `order_id`, `status`, `current_agent` |
| `POST /agents/adam` | `status`, `current_agent` |

## Setup

```bash
cd backend
cp .env.example .env          # Fill in DATABASE_URL, SECRET_KEY, etc.
pip install -r requirements.txt
alembic upgrade head          # Creates users table
uvicorn app.main:app --reload
```

### Environment Variables

| Variable | Purpose |
|----------|---------|
| `DATABASE_URL` | Supabase PostgreSQL (`postgresql+asyncpg://...`) |
| `SECRET_KEY` | JWT signing key |
| `OPENAI_API_KEY` / `ANTHROPIC_API_KEY` | LLM providers (mock fallback if unset) |
| `AGENT_API_KEY` | n8n machine-to-machine auth header |
| `ADMIN_EMAIL` / `ADMIN_PASSWORD` | Bootstrap admin user |
| `SLACK_BOT_TOKEN` / `SLACK_SIGNING_SECRET` | Victor approval notifications |
